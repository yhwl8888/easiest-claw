# 渠道能力回忆录（飞书 / Telegram）

更新时间：2026-03-20

## 背景

本轮目标聚焦两件事：

1. 渠道管理页面先把飞书、Telegram做扎实。
2. 先落地多账号和多 Agent 路由，不做其它渠道扩展。

## 已完成

### 1) 渠道状态探测链路

- 新增 `channels.status` 网关调用白名单。
- 新增 IPC：
  - `openclaw:channels:status`
- 页面支持“探测状态”，并展示当前渠道连接状态卡片。

### 2) 飞书 / Telegram 基础配置增强

- 飞书：
  - `allowFrom` / `groupAllowFrom`
  - `groups`（JSON）
  - `connectionMode` + webhook 参数
  - `streaming` / `blockStreaming`
  - `typingIndicator` / `resolveSenderNames`
- Telegram：
  - `botToken` / `tokenFile`
  - `allowFrom` / `groupAllowFrom`
  - `groups`（JSON）
  - `streamMode` / `blockStreaming`
  - webhook 参数

### 3) 多账号支持（本轮重点）

- 渠道页新增：
  - `defaultAccount`
  - `accounts JSON`
- 保存逻辑支持：
  - 直接写入 `channels.<channel>.accounts`
  - 支持单账号输入与多账号 JSON 并存（多账号 JSON 优先）

### 4) 多 Agent 路由支持（本轮重点）

- 新增 IPC：
  - `openclaw:bindings:get`
  - `openclaw:bindings:set`
- 渠道页新增：
  - `bindings JSON`（按渠道过滤编辑）
- 保存逻辑支持：
  - 仅替换当前渠道（`feishu` 或 `telegram`）对应的 `bindings` 条目
  - 保留其它渠道已有路由配置

## 本轮关键设计决策

1. 多账号与多 Agent 路由采用 JSON 编辑器先落地，优先保证能力完整和可落地。
2. `bindings` 按渠道分片保存，避免覆盖其它渠道规则。
3. 对 `dmPolicy=open` 与 `allowFrom` 做了 `*` 校验，减少配置误用。

## 仍欠缺（后续排期）

### 高优先级

1. 多账号可视化编辑器（增删账号、账号状态、字段级校验），替代 JSON 手填。
2. 多 Agent 路由可视化编辑器（规则列表、条件编辑、排序、冲突提示）。
3. 飞书/Telegram 表单文案全量 i18n（当前新增字段仍有硬编码中文）。

1. 优先做“多账号可视化编辑器”。
2. 紧接着做“多 Agent 路由可视化编辑器”。
3. 目标是替代当前 JSON 手填交互，同时保留高级 JSON 入口作为兜底。

## 风险与注意事项

1. `bindings` 是顶层全局配置；人工编辑时要避免写入其它渠道条目。
2. 多账号 JSON 与单账号输入同时使用时，以多账号 JSON 为准。
3. 若网关未连接，状态探测会失败，但本地配置读写不受影响。

3. 保存后检查 `~/.openclaw/openclaw.json` 的 `channels` 与 `bindings` 是否按预期更新。
